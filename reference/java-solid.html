S — Single Responsibility

A class should have one reason to change. Mixed responsibilities (data + presentation + persistence) → split into focused classes.

// before: one class, three jobs
class Invoice { double total(); void print(); void saveToDatabase(); }

// after: one job each
class Invoice { double total(); }
class InvoicePrinter { void print(Invoice i); }
class InvoiceRepository { void save(Invoice i); }

Heuristic: "What would force me to edit this class?" If the answer contains "and", split it.

Why it matters

Focused classes are easier to read, test, and change without side effects. The data/presentation/persistence split is the backbone of how Spring apps are organised (controller / service / repository).

O — Open/Closed

Open for extension, closed for modification. Add new behaviour by writing new classes, not by editing existing ones. A growing if/else on a "type" is the classic violation; an interface is the fix.

// before: every new channel edits this method
if (type.equals("email")) {...} else if (type.equals("sms")) {...}

// after: new channel = new class, nothing else edited
interface Channel { void send(String msg); }
class EmailChannel implements Channel { ... }
class SmsChannel   implements Channel { ... }   // add PushChannel later, no edits

Enabled by: interfaces + polymorphism (Lessons 8, 12).

L — Liskov Substitution

A subtype must be usable anywhere its parent is expected, honouring the parent's promises — not just matching signatures.

// violation: Ostrich is-a Bird but can't keep the fly() promise
class Ostrich extends Bird { void fly() { throw new UnsupportedOperationException(); } }

// fix: only fliers promise to fly
class Bird { void eat() {} }
interface Flyer { void fly(); }
class Sparrow extends Bird implements Flyer { public void fly() {} }
class Ostrich extends Bird { }   // no false promise

Tell-tale sign: an override that throws "unsupported" or does nothing contradicts the parent — a Liskov alarm.

I — Interface Segregation

No class should be forced to implement methods it doesn't use. Split fat interfaces into small, role-focused ones; a class implements only what it can honour (and can implement several).

// fat: SimplePrinter forced to fake scan()/fax()
interface MultiFunctionDevice { void print(); void scan(); void fax(); }

// segregated: implement only what applies
interface Printer { void print(); }
interface Scanner { void scan(); }
class SimplePrinter implements Printer { public void print() {} }
class AllInOne implements Printer, Scanner { ... }

Smell: methods that throw "unsupported" (also a Liskov alarm) — the interface is too big.

D — Dependency Inversion

Depend on abstractions, not concrete classes; have dependencies injected (passed in) rather than created with new inside.

// before: welded to a concrete class
class NotificationService { EmailSender sender = new EmailSender(); }

// after: depends on an interface, receives the concrete one
interface MessageSender { void send(String m); }
class NotificationService {
    private final MessageSender sender;
    NotificationService(MessageSender sender) { this.sender = sender; }  // injected
}

Payoff: swappable implementations, testable with fakes. This is what Spring automates — it creates objects and injects their dependencies for you.
