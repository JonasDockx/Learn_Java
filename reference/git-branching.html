Branching, Remotes & Undo

The team workflow, plus how to fix mistakes. Print this.

Branching

CommandDoes
git switch -c namecreate a branch and move onto it
git switch namemove onto an existing branch
git branchlist branches (current one starred)
git merge namefold name into the current branch
git branch -d namedelete a merged branch

Remotes (GitHub)

CommandDoes
git remote add origin <url>link a remote named origin
git push -u origin mainfirst push (remembers the link)
git push / git pullsend / receive commits thereafter
git clone <url>copy a remote repo down fresh

The team loop

branch → commit → push branch → open Pull Request → review → merge. The PR is where teammates review your code before it joins main.

Fixing mistakes

SituationFix
Staged a file by mistake (not committed)git restore --staged <file>
Discard uncommitted edits to a filegit restore <file>
Wrong message / forgot a file — not pushedgit commit --amend
Undo last commit, keep the editsgit reset --soft HEAD~1
Already pushed itgit revert <commit> (new undo commit)
The hard line: before pushing, fix freely; after pushing, prefer revert — don't rewrite shared history.

add . vs naming files

git add Dog.java stages one file → focused commits. git add . stages everything changed → convenient but can sweep in surprises. git status first, always.
