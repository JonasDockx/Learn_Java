TDD & JUnit

The red-green-refactor loop and JUnit basics. Print this.

The loop

RED      write a failing test for new behaviour
GREEN    write the simplest code to pass it
REFACTOR clean the code; tests stay green
↻        repeat for the next small behaviour

Never write production code without a failing test asking for it. Always green before moving on.

A JUnit test

import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

public class CalculatorTest {
    @Test
    void addsTwoNumbers() {
        Calculator calc = new Calculator();
        assertEquals(5, calc.add(2, 3));   // (expected, actual)
    }
}
  • @Test — marks a method as a test.
  • Test classes live in src/test/java; run with mvn test.

Common assertions

AssertionPasses when
assertEquals(exp, act)values are equal
assertTrue(c) / assertFalse(c)condition is true / false
assertNull(x) / assertNotNull(x)x is null / not null
assertThrows(E.class, () -> ...)the lambda throws E

Reading results

Tests run: 2, Failures: 0   → all green
Tests run: 2, Failures: 1   → one assertion failed (read which)
A failure message shows expected: <5> but was: <-1> — exactly what went wrong.
