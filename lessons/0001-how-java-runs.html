How Java Runs

From source to bytecode to the JVM — the one mental shift that separates Java from Python.

Why this, first? You already program in Python. The fastest way to start thinking in Java is to understand what happens between you saving a file and the program running — because unlike Python, there's a compile step in the middle, and it changes how you work every single day on the job.

The shift from Python

When you run Python, the interpreter reads your source and executes it directly — you save app.py, you run python app.py, done. Java inserts a step. You don't run your source code; you run a compiled version of it. 

  Python:   app.py  ───────────────▶  python runs it

  Java:     App.java  ──[ javac ]──▶  App.class  ──[ java ]──▶  JVM runs it
            (source)    compile        (bytecode)    launch

Three nouns to lock in:

  • Source — the .java file you write, human-readable.
  • Bytecode — the .class file javac produces. Not machine code, not readable — an intermediate the JVM understands.
  • JVM (Java Virtual Machine) — the program that executes bytecode. It's why Java is "write once, run anywhere": the same .class file runs on any machine with a JVM.

Your first compile-and-run

Create a file called Hello.java. The filename must match the class name — that's a Java rule, not a suggestion.

public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, Java");
    }
}

Then, in a terminal, two commands:

javac Hello.java     # compile  → produces Hello.class
java Hello           # run      → prints: Hello, Java

The key insight

Notice you compile Hello.java (with the extension) but run Hello (no extension, no .class). You're handing the class name to the JVM and asking it to find and execute that class's main method. main is always the entry point.

Why the compile step earns its keep

That extra step is the payoff of Java's static typing. In Python, a mistake like len(5) or adding a number to a string blows up only when that line actually runs — a TypeError at runtime. In Java, the compiler refuses to produce a .class file at all if the types don't line up — so a whole category of bugs is caught at javac time, before the program ever runs. On a team, that's fewer things breaking in production. We'll go deep on types in Lesson 2.

Do this now

  1. Install a JDK if you haven't — verify with java -version and javac -version in a terminal.
  2. Create Hello.java, compile it, run it. Watch the .class file appear.
  3. Break it on purpose: change println to printline and run javac again. Read the error. That's the compiler doing its job.
