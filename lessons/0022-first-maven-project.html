Your First Maven Project

One idea: a standard folder layout + a pom.xml, and Maven builds, tests, and manages libraries for you.

Until now: javac OneFile.java by hand. That doesn't scale to many files and external libraries. Maven is the tool that does: it knows where your code lives, downloads the libraries you declare, compiles, and runs your tests — with one command. I've scaffolded a working project at hello-maven/ for you to explore.

First: open a new terminal

Maven was just added to your PATH, so a terminal opened before the install won't see it. Open a fresh one (in VS Code: trash the current terminal, open a new one), then cd hello-maven. Check it with mvn -version.

The standard layout

Every Maven project has the same shape. That predictability is the point — any Java developer can open any Maven project and know where things are:

hello-maven/
├── pom.xml                         ← the project's configuration
├── src/
│   ├── main/java/                  ← your real code
│   │   └── Calculator.java
│   └── test/java/                  ← your tests (mirrors main/)
│       └── CalculatorTest.java
└── target/                         ← build output (auto-generated, gitignored)
  • src/main/java — production code. Calculator.java lives here.
  • src/test/java — test code, kept separate from the app. CalculatorTest.java lives here.
  • target/ — where Maven puts compiled classes and packaged jars. Never edit it; never commit it.

The pom.xml — the project's brain

Open hello-maven/pom.xml. The parts that matter today:

  • Coordinates (groupId, artifactId, version) — the project's unique name, like a package + project + version.
  • <dependencies> — external libraries you want. We declared JUnit (the test framework). Maven downloads it for you — no manual jar hunting.
  • <build><plugins> — tools that run during the build; here, the one that runs tests.

You declare what you need; Maven fetches it (into a local cache at ~/.m2) and wires it up. That's "dependency management".

The commands you'll use

CommandDoes
mvn compilecompile src/main/java
mvn testcompile everything and run the tests
mvn packagebuild a runnable .jar in target/
mvn cleandelete target/ (the clean build you met in Lesson 11)

Run them from the project folder (the one containing pom.xml).

What a passing run looks like

[INFO] Tests run: 1, Failures: 0, Errors: 0, Skipped: 0
[INFO] BUILD SUCCESS
BUILD SUCCESS with zero failures = everything compiled and every test passed.

Check yourself

Where does production code go in a Maven project?
What does declaring a <dependency> in pom.xml do?
Which command compiles and runs the tests?

Do this now (in the new terminal, inside hello-maven/)

  1. mvn test — confirm Tests run: 1, Failures: 0 and BUILD SUCCESS.
  2. Make the test fail on purpose: open src/main/java/Calculator.java and change a + b to a - b. Run mvn test again — watch it report a failure (expected 5, but was -1) and BUILD FAILURE. This is a test catching a bug.
  3. Change it back to a + b and confirm it passes again.
  4. Run mvn clean, look at how target/ disappears, then mvn test rebuilds it.
