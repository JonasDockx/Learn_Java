Generics: What <> Really Means

One idea: <> is a type you plug into a placeholder the class was written with.

You've written ArrayList<String> and HashMap<String, Integer>. This lesson shows where that <String> goes and why it exists — by writing your own tiny class that uses it. After this, <> stops being magic.

The idea: a type left blank, filled in later

ArrayList was written without committing to what it holds. Its author left a placeholder. When you write ArrayList<String>, you fill that placeholder with String — so this particular list holds Strings, and the compiler enforces it.

The placeholder has a name. By convention it's a single capital letter, often T (for "Type"):

class Box<T> {          // T is a placeholder for "some type, chosen later"
    private T item;     // a field of that type
    public void put(T item) { this.item = item; }
    public T get() { return item; }
}

Now T becomes whatever you put in the <> when you create one:

Box<String> s = new Box<>();   // every T in Box becomes String
s.put("hello");
String text = s.get();          // comes back as String — no cast needed

Box<Integer> n = new Box<>();  // every T becomes Integer
n.put(42);
int number = n.get();           // comes back as Integer/int

The key insight

Box<T> is written once but works for any type, and stays type-safe: a Box<String> only accepts Strings and only returns Strings. That's exactly what ArrayList<String> is — a list class written with a placeholder, which you filled with String.

Why it exists: safety without duplication

Generics solve two problems at once:

  • No duplication — you don't write a StringBox, an IntBox, a DogBox… one Box<T> covers all.
  • Type safety at compile time — Box<String> rejects a number when you compile, not as a surprise crash later. The compiler also knows get() returns a String, so you never cast.

Python comparison: a Python list holds anything and types are checked never (or only by optional, unenforced hints like list[str]). Java pushes that type into the container and the compiler enforces it — generics are how static typing reaches inside collections.

Naming convention (you'll see these)

LetterUsually means
TType (generic)
EElement (used by List<E>)
K, VKey, Value (used by Map<K,V>)

They're just names for the placeholder — like a parameter name, but for a type instead of a value.

Do this now (one file, small steps)

  1. Put Box<T> and a main in one file BoxIntro.java (a public class BoxIntro with main, plus class Box<T> below). Create a Box<String>, put text in, get it out, print it.
  2. Create a Box<Integer>, put 42 in, get it out, print it. One class, two types.
  3. On your Box<String>, try s.put(42); and compile — read the type error. The placeholder is now String, so a number is rejected.
  4. Notice you never wrote a cast on get() — the compiler already knows the type. That's the payoff.
