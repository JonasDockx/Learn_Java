Clean Code

One idea: write for the human who reads it next — usually you, in three months.

Code is read far more often than it's written. "Clean code" is code optimised for the reader, and on a team it's the difference between a change taking five minutes or an afternoon. There's no compiler check for it — it's judgment. Here are the three highest-impact habits.

Habit 1 — Intention-revealing names

A name should say what and why, so the reader doesn't have to decode the logic.

// unclear
public double c(double p, int d) {
    return p - (p * d / 100.0);
}

// clean — the names tell the story
public double applyDiscount(double price, int discountPercent) {
    return price - (price * discountPercent / 100.0);
}

Same code, but the second version explains itself. Naming is the cheapest, highest-impact clean-code habit.

Habit 2 — No magic numbers

A bare number in the middle of logic (a "magic number") hides its meaning. Give it a named constant.

// magic number — what is 18?
if (age >= 18) { allowEntry(); }

// named — the meaning is explicit
private static final int MINIMUM_AGE = 18;   // a constant
if (age >= MINIMUM_AGE) { allowEntry(); }

New keyword: final

final means the value can't be reassigned — it's a constant. Convention: constants are named in UPPER_SNAKE_CASE. static final together = "one shared, unchangeable value for the whole class". Naming the number also means you change it in one place.

Habit 3 — Small functions that do one thing

If a method does several things, split it. Each method should be describable in one short sentence — and its name should be that sentence.

// does two things: validates AND saves
void register(User u) {
    if (u.getEmail() == null) throw new IllegalArgumentException("no email");
    if (u.getAge() < 18)      throw new IllegalArgumentException("too young");
    database.save(u);
}

// split: each method does one thing, names say what
void register(User u) {
    validate(u);
    database.save(u);
}
private void validate(User u) {
    if (u.getEmail() == null) throw new IllegalArgumentException("no email");
    if (u.getAge() < 18)      throw new IllegalArgumentException("too young");
}

A note on comments

Prefer code that doesn't need a comment to explain what it does — a good name beats a comment. Reserve comments for why (a non-obvious decision, a workaround). A comment that just restates the code is noise.

Python comparison: same culture as PEP 8 / "readability counts". The Java conventions differ in spelling — camelCase for methods/variables, UPPER_SNAKE_CASE for constants, PascalCase for classes.
