Anatomy of a Java Program

No new material — just finally explaining the line you've typed ten times: public static void main.

A deliberate step back. We moved fast through objects. Before going further, let's make the structure of a Java file rock-solid — because every program you'll ever write sits inside it. You already know every piece here; we're just connecting them.

The smallest complete program

public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello");
    }
}

Two nested boxes: a class box, and inside it a method box. That's the shape of everything in Java — code always lives inside a method, which lives inside a class.

Decoding public static void main(String[] args)

The JVM starts your program by calling one special method. This is it. Word by word:

WordMeaning
publicThe JVM is outside your class and must be allowed to call this — so not private. (Lesson 7.)
staticBelongs to the class, not an object. At startup no object exists yet, so main must run without new.
voidReturns nothing. (Compare getAge(), which returns an int.)
mainThe exact name the JVM looks for. The front door must be named main.
String[] argsAn array of Strings: command-line arguments. java Hello a b → args = ["a","b"]. Usually ignored.

The one idea to keep: static = belongs to the class

Dog.main(...)          // static: called on the CLASS, no object needed
new Dog("Rex").speak() // not static: called on an OBJECT you created first
main is static because it has to run before any object exists. Object methods like speak() are not static — they need a specific Dog to act on. That single distinction is the heart of this lesson.

Why a class and its test live in separate files

Remember Lesson 7: private protects a class from other classes. So to truly test that protection, the test has to be a separate class — that's why we put main in its own file (like DogDemo) rather than inside Dog. Same reason real projects keep Dog.java and a separate DogTest.java.
