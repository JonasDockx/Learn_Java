TDD with JUnit

One idea: write the test first, watch it fail, then write just enough code to make it pass.

Test-Driven Development is a headline skill on your syllabus and a daily habit on real Java teams. The flip feels backwards at first — test before code — but it forces you to decide what "correct" means before you build, and leaves you a safety net forever after. You'll do it in hello-maven.

The loop: Red → Green → Refactor

StepDo
RedWrite a test for behaviour that doesn't exist yet. Run it — it fails (red).
GreenWrite the simplest code that makes the test pass. Run it — it passes (green).
RefactorClean up the code now that it works. Re-run tests — still green, so you didn't break anything.

Then repeat for the next small piece of behaviour. Small loops, always green before moving on.

JUnit essentials

import org.junit.jupiter.api.Test;
import static org.junit.jupiter.api.Assertions.*;

public class CalculatorTest {
    @Test                                  // marks a test method
    void addsTwoNumbers() {
        Calculator calc = new Calculator();
        assertEquals(5, calc.add(2, 3));   // assertEquals(expected, actual)
    }
}

The key assertions (the checks that make a test pass or fail):

AssertionPasses when
assertEquals(exp, act)act equals exp
assertTrue(cond)cond is true
assertThrows(E.class, () -> ...)the code throws E

Python comparison: JUnit's @Test void foo() + assertEquals is pytest's def test_foo(): assert ...; assertThrows is pytest.raises.

A full cycle, worked: add subtract

① Red — add this test to CalculatorTest.java (the method doesn't exist yet):

@Test
void subtractsTwoNumbers() {
    Calculator calc = new Calculator();
    assertEquals(2, calc.subtract(5, 3));
}

Run mvn test → it fails to compile ("cannot find symbol: method subtract"). That's red — the test is demanding behaviour that isn't there.

② Green — add the method to Calculator.java:

public int subtract(int a, int b) {
    return a - b;
}

Run mvn test → green. Tests run: 2, Failures: 0.

③ Refactor — look at the code; if it could be cleaner, tidy it now, knowing the tests will catch any mistake. (Here it's already minimal, so nothing to do — but the habit of pausing to clean up is the point.)

The key insight

The test came first, and its failure drove the code you wrote. You never write production code without a failing test asking for it. The payoff compounds: every cycle adds a permanent check, so you can change code later and instantly know if you broke something.

Do this now (in hello-maven, the TDD loop yourself)

  1. Red: in CalculatorTest.java, add a test for a multiply method that doesn't exist yet: assertEquals(12, calc.multiply(3, 4));. Run mvn test — confirm it fails.
  2. Green: add multiply to Calculator.java. Run mvn test — confirm all tests pass.
  3. Refactor: nothing needed — just re-read your code and note it's clean. The habit matters more than the edit.
  4. Stretch (assertThrows): write a test that calc.divide(1, 0) throws: assertThrows(ArithmeticException.class, () -> calc.divide(1, 0));. Watch it fail, then add a divide method that throws an ArithmeticException when b == 0. Green again.
