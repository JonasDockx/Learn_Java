Encapsulation

Stop letting outside code reach in and break your objects. The first OO pillar — and the one you'll apply every single day.

Fixing a hole you already saw. In Lesson 4, anyone could write dog.age = 4 — or dog.age = -100. Nothing stopped them. Real classes guard their own data. Encapsulation is how, and it's the design habit interviewers and code reviewers look for first in a junior.

The problem with public fields

Dog d = new Dog("Rex", 3);
d.age = -100;          // nonsense — but the compiler allows it
d.name = null;         // now bark() will crash later

The fields are exposed. Any code anywhere can put the object into an invalid state, and you'll never know where the bad value came from. The fix: make the data private, and force all access through methods you control.

private fields + getters and setters

public class Dog {
    private String name;     // private = unreachable from outside the class
    private int age;

    public Dog(String name, int age) {
        this.name = name;
        setAge(age);         // reuse the validation even at construction
    }

    public String getName() {        // getter: controlled read access
        return name;
    }

    public int getAge() {
        return age;
    }

    public void setAge(int age) {    // setter: controlled write access
        if (age < 0) {
            throw new IllegalArgumentException("age cannot be negative");
        }
        this.age = age;              // only reached if the value is valid
    }
}

The key insight

private is enforced by the compiler — d.age = -100 now fails to build, not at runtime, never. The object can only be changed through setAge, which refuses invalid values. The data and the rules that protect it live together in one class. That bundling-with-a-guarded-door is encapsulation.

The Python contrast

Python relies on convention — a leading underscore (self._age) politely signals "private," but nothing stops you touching it; the culture is "we're all adults here." Java enforces it: private is a hard wall the compiler guards. Python's @property is the closest analog to getters/setters — same goal (run code on access), different mechanism.

Do this now

  1. Rewrite your Dog.java: make name and age private, add getName(), getAge(), and a validating setAge().
  2. In main, try d.age = -100; and compile — read the "has private access" error. Delete that line.
  3. Now try d.setAge(-100); and run — watch your IllegalArgumentException fire with your message. You defined that rule.
  4. Use d.getAge() to print the age the proper way. Commit it on a branch and merge (practise Lesson 6!).
