Abstract Classes

One idea: a class you can extend but cannot instantiate.

This is the abstraction topic from before, narrowed to a single concept and a single file. No interfaces here — those come later.

The idea

An abstract class exists only to be a parent. Two rules:

  1. You cannot create one with new — only its subclasses.
  2. An abstract method has no body. Any subclass must provide one, or it won't compile.

Use it when a base type is real as a category but meaningless on its own. "A vehicle" makes no sense as a concrete thing — only a car, a bike. So Vehicle is abstract; Car is concrete.

One file, three classes

A .java file may hold one public class plus other non-public classes. Put this all in AbstractIntro.java:

public class AbstractIntro {
    public static void main(String[] args) {
        // Vehicle v = new Vehicle();   // ← uncomment later to see the error
        Car car = new Car();
        System.out.println(car.sound());   // vroom
    }
}

abstract class Vehicle {        // cannot be instantiated
    abstract String sound();    // no body — subclasses must implement
}

class Car extends Vehicle {     // concrete: provides the body
    @Override
    String sound() {
        return "vroom";
    }
}

Compile and run:

javac AbstractIntro.java
java AbstractIntro

It prints vroom. Car works because it supplies sound().

See rule 1 fail on purpose

Uncomment the Vehicle v = new Vehicle(); line and recompile. You get: 
error: Vehicle is abstract; cannot be instantiated
That is the entire point of abstract: it makes "a generic Vehicle" impossible to build. (This error appears even from inside the same file — unlike private, which only blocks other classes.)

See rule 2 fail on purpose

Temporarily delete the whole sound() method from Car and recompile. You get:

error: Car is not abstract and does not override abstract method sound() in Vehicle

Translation: "Car claims to be a real, buildable vehicle, but it never said what sound it makes." A concrete subclass must fill in every abstract method. Put sound() back afterward.

If that error does not appear: stale .class files

Compiled .class files persist in the folder, and javac resolves a class name against any .class it finds — not just your current source. A leftover Car.class from an earlier build can silently satisfy new Car() even after you change the source, hiding the error. Fix: a clean build — delete the .class files and recompile: 
del *.class      && javac AbstractIntro.java    # Windows
rm *.class       ;  javac AbstractIntro.java     # Git Bash
Real projects automate this with mvn clean / gradle clean.

Do this now (one file, small steps)

  1. Create AbstractIntro.java with the code above. Compile and run — confirm it prints vroom.
  2. Uncomment new Vehicle(), recompile, read the "cannot be instantiated" error, then comment it back.
  3. Delete Car's sound() method, recompile, read the "must override" error, then put it back.
  4. Add a second subclass Bike that returns "ring", create one in main, and print its sound.

That is the whole lesson: cannot instantiate the abstract parent; must implement its abstract methods.
