Dependency Inversion Principle

One idea: depend on an interface, and have the concrete thing handed to you — don't build it yourself.

The "D" of SOLID, and the most important for where you're headed: this is the principle Spring is built on. Master the shape here and Spring's "dependency injection" will look familiar instead of magical. It ties together interfaces (12), Open/Closed (28), and testability (TDD).

The problem: welded to a concrete class

class EmailSender {
    void send(String message) { /* ...email... */ }
}

class NotificationService {
    private EmailSender sender = new EmailSender();   // ← creates its own dependency

    void alert(String message) {
        sender.send(message);
    }
}

NotificationService is welded to EmailSender: it names the concrete class and builds it with new. You can't switch to SMS without editing this class, and you can't test it without really sending emails. The high-level service depends directly on a low-level detail.

The fix: depend on an abstraction, receive the concrete one

interface MessageSender {                 // the abstraction
    void send(String message);
}

class EmailSender implements MessageSender {
    public void send(String message) { /* ...email... */ }
}
class SmsSender implements MessageSender {
    public void send(String message) { /* ...SMS... */ }
}

class NotificationService {
    private final MessageSender sender;

    NotificationService(MessageSender sender) {   // ← handed in, not created here
        this.sender = sender;
    }

    void alert(String message) {
        sender.send(message);                     // works with ANY MessageSender
    }
}

Now the caller decides which sender to use and passes it in:

var service = new NotificationService(new EmailSender());   // or new SmsSender()
service.alert("Hello");

The key insight — the "inversion"

Before, NotificationService controlled its own dependency (new EmailSender()). Now it just declares "I need some MessageSender" and something outside supplies it through the constructor. Control of the dependency is inverted — handed to the caller. Passing a dependency in like this is called dependency injection.

Why this is a big deal

  • Swappable: pass SmsSender instead of EmailSender — no change to NotificationService.
  • Testable: in a test, pass a fake MessageSender that just records the message — no real emails. This is how you unit-test with TDD.
  • This is Spring: Spring automatically creates your objects and injects the right dependencies into their constructors. DIP is the manual version of what Spring does for you — which is why it comes right before the Spring block.

Python comparison: exactly the "pass collaborators in rather than hardcode them" pattern; Java + Spring just automate the wiring.
