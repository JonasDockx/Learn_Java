Branching & GitHub

Work on a feature without touching the main line — then push it somewhere the team (and your future job) can see it.

This is the team workflow. On the job you won't commit straight to the main branch. You'll branch off, build a feature, push it to GitHub, and open a pull request for review. This lesson walks that exact loop on your own repo.

What a branch is

A branch is just a movable pointer to a commit — a parallel line of work. You make a branch, commit on it freely, and your main branch stays untouched until you decide to merge the work back in. It's how a team works on ten things at once without stepping on each other. 

  main:     A───B───C
                     \
  feature:            D───E      ← your work, isolated
                     /
  (merge E back into main when it's ready)

The branching loop

git switch -c add-cat-class   # create a branch AND move onto it
git status                    # confirms: "On branch add-cat-class"

# ...edit files, then the normal add/commit loop...
git add Cat.java
git commit -m "Add Cat class"

git switch main               # hop back to the main line
git merge add-cat-class       # fold the feature work into main
git branch -d add-cat-class   # delete the branch once merged

The key insight

git switch changes which branch you're "standing on" — your files in the folder actually change to match that branch. Commits you make land on the current branch only. main doesn't see them until you merge. (You may see older guides use git checkout; git switch is the modern, clearer verb.)

GitHub: a remote home for your repo

So far everything lives only on this laptop. GitHub hosts a copy — a remote — so your work is backed up, shareable, and reviewable. The vocabulary:

TermMeaning
remotea linked copy of your repo elsewhere (e.g. on GitHub). The default is named origin.
pushsend your local commits up to the remote
pullbring the remote's new commits down to you
clonecopy a remote repo down for the first time
pull request (PR)a GitHub request to merge your branch — where code review happens

Connecting your existing repo to a fresh empty GitHub repo (one-time):

git remote add origin https://github.com/<you>/Learn_Java.git
git push -u origin main        # push main; -u remembers the link for next time

After that, day to day it's just git push and git pull.

The team loop you'll actually live in

branch → commit → push branch → open a Pull Request → review → merge. The PR is the heart of it: teammates comment on your code before it joins main. "Can you open a PR?" is junior-day-one vocabulary — now you have it.

Do this now — part A (local, do it today)

  1. git switch -c add-cat-class. Run git status to see the branch name.
  2. Create a small Cat.java (copy your Dog.java shape). Compile, git add, git commit -m "Add Cat class".
  3. git switch main — open the folder and notice Cat.java is gone (it lives on the other branch).
  4. git merge add-cat-class — now Cat.java is back on main. git log --oneline to see it.
  5. git branch -d add-cat-class to clean up.

Do this now — part B (GitHub, when you're ready)

Needs a free GitHub account and one-time authentication — a slightly bigger setup, so do it when you have a quiet 20 minutes. Ask me and I'll walk you through it.

  1. Create a (free) account at github.com.
  2. Make a new empty repo named Learn_Java (no README — yours isn't empty).
  3. Run the two git remote add / git push -u origin main commands above.
  4. Refresh the GitHub page — your code is now online.
