CI/CD

One idea: a server automatically builds and tests your code every time you push.

You run mvn test by hand. Continuous Integration (CI) makes a server do it for you on every push — so a broken test is caught immediately, for everyone, not just when someone remembers to check. It's the automation that ties together everything in this block: tests, build, the whole safety net.

CI and CD

  • CI — Continuous Integration: on every push, an automated server checks out your code, builds it, and runs the tests. If anything fails, you (and the team) are told at once.
  • CD — Continuous Delivery/Deployment: the next step — after tests pass, automatically package and ship the app. As a junior you'll mostly live in CI; CD comes once there's something to deploy.

Today is CI: your tests, run automatically on GitHub.

How it works: a workflow file

GitHub runs CI through GitHub Actions. You describe the steps in a YAML file inside your repo, at .github/workflows/. GitHub watches for it and runs it on its own servers. I've created one for you — open .github/workflows/ci.yml. 

name: CI
on:
  push:               # run on every push
  pull_request:       # and every PR

jobs:
  build-and-test:
    runs-on: ubuntu-latest        # a fresh Linux machine
    steps:
      - uses: actions/checkout@v4         # 1. get the code
      - uses: actions/setup-java@v4       # 2. install Java 21
        with: { distribution: temurin, java-version: '21' }
      - run: mvn -B test                  # 3. run the tests
        working-directory: hello-maven

Read it as three steps on a fresh machine: get the code → install Java → run mvn test. Exactly what you do locally, automated.

YAML — and a Python parallel

.yml is YAML, a config format that, like Python, uses indentation to show structure (nesting by spaces, no braces). name:, on:, jobs: are keys; the indented lines under them belong to them. Watch your spaces — YAML is as indentation-sensitive as Python.

What you'll see

After you push this, GitHub shows an Actions tab on your repo. Each push gets a run with a green check (passed) or a red X (failed). A failing test turns the check red and emails you — the bug is caught the moment it lands, not days later.

Do this now

  1. Make sure your hello-maven project and the new .github/workflows/ci.yml are committed. From the repo root: git add ., git commit -m "Add Maven project and CI workflow".
  2. git push. Then open your repo on GitHub and click the Actions tab — watch the workflow run the three steps and finish with a green check.
  3. See CI catch a bug: break a test (change Calculator.add to a - b), commit, and push. Watch the Actions run go red. Then fix it, push again, watch it go green.
