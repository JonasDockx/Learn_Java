Refactoring

One idea: improve the code's structure without changing what it does — with tests proving you didn't break it.

Refactoring is the "R" in red-green-refactor. It means cleaning up working code — better names, less duplication, smaller methods — while its behaviour stays identical. This is why you wrote tests first: they're the safety net that lets you change code fearlessly.

The definition (and the rule)

Refactoring = changing the internal structure of code without changing its external behaviour. The hard rule: the tests that passed before must still pass after. If behaviour changes, that's not refactoring — that's a new feature or a bug.

The loop is tiny: make one small improvement → run the tests → still green → continue. Green after every step means you're never more than one undo away from safety.

A real refactor: remove duplication in your tests

Look at your CalculatorTest.java. Every test starts with the same line:

@Test void addsTwoNumbers()      { Calculator calc = new Calculator(); ... }
@Test void subtractsTwoNumbers() { Calculator calc = new Calculator(); ... }
@Test void multiplyTwoNumbers()  { Calculator calc = new Calculator(); ... }
// ...the same `new Calculator()` repeated in every test

That repetition is duplication — a classic refactoring target ("DRY": Don't Repeat Yourself). JUnit gives us a tool to remove it: a method marked @BeforeEach runs automatically before every test.

import org.junit.jupiter.api.BeforeEach;     // add this import

public class CalculatorTest {

    Calculator calc;                  // one field, shared by the tests

    @BeforeEach
    void setUp() {
        calc = new Calculator();      // runs fresh before EACH test
    }

    @Test
    void addsTwoNumbers() {
        assertEquals(5, calc.add(2, 3));    // no more `new Calculator()` here
    }
    // ...the other tests lose their duplicated line too
}

The key insight

After this change, run mvn test — still all green. You changed the code's structure (removed five duplicated lines, added a setup method) but not its behaviour: the same tests check the same things. That "still green" is the proof that makes refactoring safe. Without tests, you'd be guessing.

Common refactorings

RefactoringWhat
Renamegive a variable/method a clearer name
Extract methodpull a chunk of code into its own well-named method
Remove duplication (DRY)one source of truth instead of copy-paste
Inline / simplifydelete needless indirection or complexity

Python comparison: refactoring is language-agnostic; @BeforeEach is JUnit's version of a pytest fixture that prepares fresh state for each test.

Do this now (in hello-maven)

  1. Run mvn test first and note the green count — that's your baseline.
  2. Refactor CalculatorTest.java: add the @BeforeEach import, add a calc field and a setUp() method, and delete the repeated Calculator calc = new Calculator(); from each test.
  3. Run mvn test again — confirm the same green count. Same behaviour, cleaner code. That's a successful refactor.
  4. One more: rename a test to describe its behaviour better (e.g. multiplyTwoNumbers → multipliesTwoNumbers). Re-run — still green.
