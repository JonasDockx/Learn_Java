Meet Spring: Injection, Automated

One idea: Spring creates your objects and injects their dependencies for you — the DIP you just did by hand.

You just wrote dependency injection manually: new NotificationService(new EmailSender()). Spring does exactly that wiring automatically, for a whole application. Its core is an IoC container — a manager that builds your objects and plugs them into each other. I've scaffolded a tiny Spring app at spring-demo/ that proves it. This is the gateway to enterprise Java.

First: run it (in a terminal, inside spring-demo/)

cd spring-demo
mvn spring-boot:run
Among Spring's start-up logs you'll see one line from your code: 
Email: Hello from Spring
Nothing in your code called new. Spring built and connected everything. Press Ctrl+C to stop.

What just happened

Spring's IoC container (Inversion of Control) took over object creation. On start-up it:

  1. Scanned your package for classes marked as components.
  2. Created one instance of each (these managed objects are called beans).
  3. Looked at each constructor, saw what it needed, and injected the matching bean.

So EmailSender was injected into NotificationService, which was injected into AppRunner — a whole chain wired automatically. That "the framework controls creation, not your code" is the Inversion of Control.

The annotations that make it work

AnnotationMeans
@SpringBootApplicationthe entry point; start the container and scan this package
@Component"Spring, manage a bean of this class" (e.g. EmailSender)
@Service@Component that reads as "business logic" (e.g. NotificationService)

The injection itself needs no annotation: Spring sees a constructor that takes a MessageSender and supplies the bean that fits. It's the constructor injection you already wrote — now automatic.

New: real packages

Notice the files start with package com.jonas.demo; and live in folders src/main/java/com/jonas/demo/. A package is a named namespace (Lesson: Java's namespace is the package). Spring scans the entry class's package and below, so real apps always use packages — the folder path must match the package name.

The key insight

You declare what your components are (@Component) and what they need (constructor parameters). Spring figures out how to build and connect them. You stop writing wiring code (new chains) and let the container do it — which is what makes big applications manageable. DIP was the principle; Spring is the machine that runs on it.

Do this now

  1. Run mvn spring-boot:run in spring-demo/. Find the Email: Hello from Spring line in the logs.
  2. Change the message in AppRunner.run(...) and re-run — confirm your new text appears.
  3. See injection swap with no wiring change: add an SmsSender class (@Component implements MessageSender) that prints "SMS: " + message. Because there'd now be two MessageSender beans, add @Primary above one of them so Spring knows which to inject. Re-run and watch the output switch — you changed a bean, not the service.
