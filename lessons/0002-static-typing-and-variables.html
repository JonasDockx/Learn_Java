Static Typing & Variables

Why every Java variable has a type stamped on it for life — and why that's the compiler working for you.

Building on Lesson 1. You saw that javac checks types before producing bytecode. This lesson is about where those types come from: you declare them. That single habit is the biggest day-to-day difference between writing Python and writing Java.

The Python contrast

In Python, a name is just a label you can stick on anything, and re-stick whenever you like:

# Python — totally fine
x = 5
x = "now I'm a string"
x = [1, 2, 3]

In Java, a variable is born with a type, and that type never changes. You declare it once, on the left:

// Java — the type comes first
int x = 5;
x = 10;            // fine — still an int
x = "hello";       // COMPILE ERROR — x is an int, forever

The key insight

In Python the value knows its type; the variable is just a label. In Java the variable has the type, fixed at declaration. The compiler uses that promise to catch mistakes — x = "hello" never even builds, so it can never blow up at runtime.

The types you'll use constantly

Java splits into primitives (lowercase, simple values) and objects (capitalised, like String). The handful you'll reach for daily:

int     count   = 42;          // whole numbers
double  price   = 19.99;       // decimals
boolean isReady = true;        // true / false  (note: lowercase!)
char    grade   = 'A';         // a single character, single quotes
String  name    = "Jonas";     // text, double quotes, capital S
  • int vs double: Python has one int and one float; Java makes you choose the size/kind up front. Start with int for whole numbers, double for decimals.
  • boolean is true/false — lowercase, unlike Python's True/False.
  • char is a single character in single quotes; String is text in double quotes. Mixing the quotes up is a common first-week error.

Let the compiler write the type: var

Modern Java (10+) can infer the type from the value, so you don't always repeat yourself:

var count = 42;        // compiler infers: int
var name  = "Jonas";   // compiler infers: String

This looks like Python, but it's not the same: count is still permanently an int — you just didn't type the word. count = "x" still won't compile. Use var when the type is obvious from the right-hand side; write the type explicitly when it adds clarity.

Check yourself

In Java, after int x = 5;, what does x = "hi"; do?

Do this now

Make a file Types.java and run it:

public class Types {
    public static void main(String[] args) {
        int count = 3;
        double price = 19.99;
        String name = "Jonas";
        boolean isReady = true;

        System.out.println(name + " has " + count + " items, $" + price);
        System.out.println("Ready? " + isReady);
    }
}
  1. javac Types.java then java Types — see the output. Notice + glues strings and numbers together into text.
  2. Now add the line count = "four"; before the print and recompile. Read the compiler error — it will name the types. That's static typing protecting you.
  3. Change one int to var and confirm it still works.
