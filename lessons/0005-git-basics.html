Git Basics

Turn this very folder into a versioned project — and make your first commits from the Java files you've already written.

Why now? You have five real .java files in this folder and no safety net. Git is that net: a series of save points you can return to, compare, and share. Every team you'll ever join runs on it, and "can you use Git" is assumed on day one of a junior role. We'll learn it on your own files, not toy examples.

The one mental model

Git work happens in three places. Almost everything you do is moving changes rightward:

  Working directory  ──[ git add ]──▶  Staging area  ──[ git commit ]──▶  Repository
  (your files,                          (changes you've               (permanent
   edited live)                          chosen for the               save points /
                                         next save point)              history)

The staging area is the part newcomers miss. It lets you pick exactly which changes go into the next commit — you might edit four files but commit only two. Think of it as the box you load before sealing it shut with commit.

One-time setup

Git needs to know who's making the commits. Set this once per machine:

git config --global user.name  "Jonas Dockx"
git config --global user.email "dockxjonas@gmail.com"

The core loop

git init               # turn the current folder into a Git repo (run once)
git status             # what's changed? (run this constantly — it's your radar)
git add Hello.java     # stage one file...
git add .              # ...or stage everything that changed
git commit -m "Add first Java practice files"   # seal a save point with a message
git log --oneline      # see your history of commits

The key habit

git status is your most-used command — run it before and after every add and commit to see exactly what Git thinks is going on. A good commit message finishes the sentence "If applied, this commit will…" — e.g. "Add Dog class demonstrating references." Present tense, specific.

Keeping junk out: .gitignore

Some files should never be committed: compiled .class files (regenerated every build), IDE settings, OS clutter, and — critically — secrets like keys and passwords. A .gitignore file lists patterns Git should pretend it can't see. I've put a ready-made one in this folder; open .gitignore and read it. 

*.class      # bytecode — never commit; javac regenerates it
target/      # build output (later lessons)
id_rsa       # an SSH private key — committing a secret = treat it as leaked
.env         # environment files with tokens/passwords
Rule that's saved many careers: a secret committed even once lives in history forever. .gitignore before the first git add, not after.

Check yourself

What does git add do?
Why list *.class in .gitignore?
Which command shows what's currently changed but uncommitted?

Do this now

In a terminal in this folder (Learn_Java):

  1. Set your name/email with the two git config commands above (once).
  2. git init — then git status. Notice your .class files are not listed (the .gitignore is already working).
  3. git add . then git status again — watch files turn green (staged).
  4. git commit -m "Add Java fundamentals practice files" — your first save point.
  5. Edit one file (add a comment), then git status, git add, and commit again with a clear message.
  6. git log --oneline — see your two commits. That's your history.
