Interface Segregation Principle

One idea: many small, focused interfaces beat one big "do everything" interface.

The "I" of SOLID — and you've already used it. When you moved fly() out of Bird into a Flyer interface so ostriches weren't forced to implement it, that was interface segregation. This lesson names the principle and its smell.

The principle

No class should be forced to implement methods it doesn't use. A big interface that bundles unrelated abilities forces every implementer to provide all of them — even the ones that make no sense for it. Split the fat interface into small, focused ones.

The smell: a fat interface

interface MultiFunctionDevice {
    void print();
    void scan();
    void fax();
}

// A simple printer can only print — but it's forced to implement all three:
class SimplePrinter implements MultiFunctionDevice {
    public void print() { /* ...ok... */ }
    public void scan()  { throw new UnsupportedOperationException(); }   // forced, fake
    public void fax()   { throw new UnsupportedOperationException(); }   // forced, fake
}

Those thrown methods are the giveaway — and notice it's also the Liskov alarm from last lesson. The fat interface caused the violation: it forced SimplePrinter to fake abilities it doesn't have.

The fix: small, focused interfaces

interface Printer { void print(); }
interface Scanner { void scan(); }
interface Fax     { void fax(); }

class SimplePrinter implements Printer {           // only what it can do
    public void print() { /* ... */ }
}

class OfficeAllInOne implements Printer, Scanner, Fax {   // implements many
    public void print() { /* ... */ }
    public void scan()  { /* ... */ }
    public void fax()   { /* ... */ }
}

The key insight

Each class implements exactly the interfaces it can honour — no fakes, no thrown "unsupported". A capable device implements several small interfaces (remember: a class can implements many). This is why interfaces should be small and role-focused, not giant catch-alls.

Why it matters

Small interfaces mean code depends only on the abilities it actually needs. A method that just prints can take a Printer — it doesn't drag in scanning and faxing it will never call. Less coupling, fewer fake methods, no Liskov violations.

Python comparison: the same instinct as small, focused Protocols rather than one huge base class.
