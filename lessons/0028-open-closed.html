Open/Closed Principle

One idea: add new behaviour by writing new code, not by editing existing code.

The "O" of SOLID: code should be open for extension, closed for modification. You should be able to add a new case without reopening and risking the code that already works. This is the payoff of the interfaces and polymorphism you learned in Lessons 12 and 8 — now as a design principle.

The smell: an if/else that grows forever

class Notifier {
    void send(String type, String message) {
        if (type.equals("email")) {
            // ...send an email...
        } else if (type.equals("sms")) {
            // ...send an SMS...
        }
        // want push notifications? EDIT this method again.
    }
}

Every new channel means reopening Notifier and adding another branch — risking the branches that already work, and forcing a re-test of the whole thing. The class is closed to easy extension and open to accidental breakage. Backwards.

The fix: an interface each new case implements

interface Channel {
    void send(String message);       // the contract
}

class EmailChannel implements Channel {
    public void send(String message) { /* ...email... */ }
}

class SmsChannel implements Channel {
    public void send(String message) { /* ...SMS... */ }
}

class Notifier {
    void notify(Channel channel, String message) {
        channel.send(message);       // doesn't care which channel — polymorphism
    }
}

The key insight

To add push notifications, you write a new class PushChannel implements Channel — and touch nothing that already exists. Notifier, EmailChannel, SmsChannel are all untouched and un-retested. That's "open for extension" (new classes) and "closed for modification" (existing code stays put). The interface is the seam that makes it possible.

Why it matters

Code you don't modify is code you can't break. Adding features by adding classes — rather than editing crowded methods — keeps existing, tested behaviour safe. It also plugs into polymorphism: Notifier works with any Channel, including ones that don't exist yet.

Python comparison: same principle; Python often reaches it through duck typing, Java through a shared interface the compiler checks.

Here's an Open/Closed violation. Refactor it to an interface-based design so a new shape needs no edit to existing code. Write the interface and two shape classes (you can keep the area formulas simple), and show how the caller uses them. Paste it in the chat.

class AreaCalculator {
    double area(String shape, double size) {
        if (shape.equals("square")) return size * size;
        else if (shape.equals("circle")) return 3.14 * size * size;
        // new shape → edit here (the smell)
    }
}

Hint: an interface Shape with a double area() method; Square and Circle implement it; the caller just calls shape.area().
