Single Responsibility Principle

One idea: a class should do one job — have one reason to change.

Design principles are guidelines for structuring code so it stays easy to change. The most famous set is SOLID — five principles, which we'll take one lesson at a time. This is the first, the "S": Single Responsibility. It's clean code's "do one thing," lifted from methods up to whole classes.

The principle

A class should have one responsibility — Robert Martin's phrasing: one reason to change. If you can describe a class with "it does X and Y and Z", those "and"s are a warning: it's holding multiple jobs, and a change to one risks breaking the others.

Before: a class doing too much

class Invoice {
    // responsibility 1: the data + business calculation
    private double amount;
    double total() { return amount * 1.21; }   // adds tax

    // responsibility 2: presentation
    void printToConsole() { System.out.println("Total: " + total()); }

    // responsibility 3: persistence
    void saveToDatabase() { /* ...SQL... */ }
}

This Invoice has three reasons to change: the tax rule, the print format, and how it's stored. A change to the database code sits right next to the business logic, easy to break by accident. It's also hard to test one part in isolation.

After: one responsibility each

class Invoice {                       // just the data + calculation
    private double amount;
    double total() { return amount * 1.21; }
}

class InvoicePrinter {                // just presentation
    void print(Invoice invoice) { System.out.println("Total: " + invoice.total()); }
}

class InvoiceRepository {             // just persistence
    void save(Invoice invoice) { /* ...SQL... */ }
}

The key insight

Now each class has one reason to change: change the tax rule → only Invoice; change the output → only InvoicePrinter; change storage → only InvoiceRepository. Smaller, focused classes are easier to understand, test, and change without surprises. (This separation — data vs. presentation vs. persistence — is exactly how Spring apps are structured, coming up later.)

The heuristic

Ask: "what would force me to edit this class?" If the answer has an "and" in it — "if the tax rule changes and if the report format changes" — it's doing too much. One reason to change = one responsibility.

Python comparison: SRP is language-agnostic — the same "one module, one job" instinct behind well-structured Python packages.

class UserAccount {
    String name;
    String email;

    boolean isValidEmail() { return email.contains("@"); }   // validation
    void saveToDatabase()  { /* ...SQL... */ }                // persistence
    void sendWelcomeEmail(){ /* ...SMTP... */ }               // emailing
}

Hint: how many different reasons could force you to edit this class? Each one is a responsibility that could become its own class.
