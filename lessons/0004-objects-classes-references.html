Objects, Classes & References

The blueprint, the instance, and the pointer between them — and finally, why == lied about your name.

The pivot. Everything so far was "Java the language." From here it's "Java the way teams actually build software" — and it all rests on classes. You know classes from Python, so we move fast on the idea and slow on two things: Java's syntax, and the reference model you bumped into last lesson.

Class vs object

A class is a blueprint; an object is a thing built from it. You know this from Python — here it is side by side:

# Python                          // Java
class Dog:                        public class Dog {
    def __init__(self, name, age):    String name;          // typed fields,
        self.name = name              int age;              // declared up front
        self.age = age

                                      Dog(String name, int age) {   // constructor
                                          this.name = name;         // = __init__,
                                          this.age = age;           // named after class
                                      }

    def bark(self):                   String bark() {       // method, typed return
        return f"{self.name} woof"        return name + " woof";
                                      }
                                  }

d = Dog("Rex", 3)                 Dog d = new Dog("Rex", 3);   // note: new

Four differences to notice:

  • Fields are declared with types at the top (String name;), not invented inside __init__.
  • The constructor is named after the class (Dog(...)), not __init__. It's the code that runs when you build an object.
  • this is Python's self — but you don't list it as a parameter. Java passes it for you.
  • new actually builds the object. new Dog("Rex", 3) = "construct a fresh Dog."

The reference model — your == answer

This is the important part. When you write:

Dog d = new Dog("Rex", 3);

d does not hold the Dog. It holds a reference — an arrow pointing to the Dog object, which lives elsewhere in memory. This is the split you discovered last lesson:

KindThe variable holds…Examples
Primitivethe value itselfint, double, boolean, char
Reference typean arrow to an objectString, Dog, arrays, everything capitalised

So when two variables point at the same object, copying one copies the arrow, not the object:

Dog a = new Dog("Rex", 3);
Dog b = a;          // copies the arrow — same Dog!
b.age = 4;
// a.age is now 4 too — a and b point at ONE object

Dog c = new Dog("Rex", 3);   // a different object, same contents
System.out.println(a == c);  // false — two different objects (arrows differ)

The key insight — == resolved

== compares the arrows: "do these point at the same object?" For primitives there's no arrow, so == compares values — which is why it works for int. For objects it compares identity, which is why name == "Jonas" only worked by the pool coincidence. .equals() exists to compare contents instead. Mystery closed.

What does new Dog("Rex", 3) do?

Do this now

Create Dog.java — one file, blueprint plus a main to try it:

public class Dog {
    String name;
    int age;

    Dog(String name, int age) {
        this.name = name;
        this.age = age;
    }

    String bark() {
        return name + " says woof";
    }

    public static void main(String[] args) {
        Dog a = new Dog("Rex", 3);
        Dog b = a;            // same object
        b.age = 4;
        System.out.println(a.name + " is " + a.age);   // age is 4 — why?
        System.out.println(a.bark());

        Dog c = new Dog("Rex", 4);   // different object, same contents
        System.out.println("a == b? " + (a == b));     // true  (same arrow)
        System.out.println("a == c? " + (a == c));     // false (different objects)
    }
}
  1. javac Dog.java, java Dog. Confirm a.age became 4 — say out loud why.
  2. Add a second method, e.g. String describe() returning name + ", age " + age, and print it.
  3. Predict each == result before running. Getting these right means the reference model has clicked.
