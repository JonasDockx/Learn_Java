Abstraction: Abstract Classes & Interfaces

Define what something must do without saying how — the contracts that hold large Java systems together.

Why this matters for the job. Spring, and most Java libraries, are built on interfaces: you code against a contract, the framework supplies the implementation. Understanding abstract classes and interfaces is what lets you read a real codebase and think "ah, this is the contract, that's the implementation." This is the abstraction pillar on your syllabus.

Abstract classes — a base that can't stand alone

From your own question: there's no such thing as a generic Animal. Mark it abstract so nobody can create one, and declare speak() with no body so every subclass must supply it:

public abstract class Animal {        // can't be instantiated
    private String name;
    public Animal(String name) { this.name = name; }
    public String getName() { return name; }

    public abstract String speak();   // no body — subclasses MUST implement
}
Animal a = new Animal("Thing");   // ✗ COMPILE ERROR — Animal is abstract
Animal d = new Dog("Rex");        // ✔ a concrete subclass is fine

An abstract class can still have state (the name field) and finished methods (getName()) — it's a partial blueprint: "here's what all animals share, plus a list of what each must fill in."

Interfaces — a pure contract

An interface goes further: it's only a list of methods a class promises to provide — a capability, with no state. A class implements it: 

public interface Trainable {        // a capability, not a thing
    String learnTrick(String trick);
}

public class Dog extends Animal implements Trainable {
    public Dog(String name) { super(name); }

    @Override public String speak() { return getName() + " says woof"; }

    @Override
    public String learnTrick(String trick) {
        return getName() + " learned to " + trick;
    }
}

The key insight — the one-vs-many rule

A class can extends exactly one class, but can implements many interfaces. So interfaces are how unrelated classes share a capability: a Dog and a Robot could both be Trainable without sharing any parent. Inheritance shares identity (is-a); interfaces share ability (can-do).

Which one? abstract class vs interface

Abstract classInterface
Modelsis-a (shared identity)can-do (shared ability)
Can hold fields/stateyesno (constants only)
How many per classextends oneimplements many
Use whensubtypes share code + stateunrelated types share a capability

Rule of thumb for now: a shared base with real shared code → abstract class; a capability many types can sign up to → interface. (Python parallels: abc.ABC + @abstractmethod for both; Java just separates them into two keywords.)
