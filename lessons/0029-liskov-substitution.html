Liskov Substitution Principle

One idea: a subclass must work anywhere its parent is expected — without breaking what callers rely on.

The "L" of SOLID. Inheritance (Lesson 8) says a Dog is-a Animal. Liskov adds a condition: the subtype must actually behave like the parent everywhere the parent is used. If code works with an Animal but breaks when you hand it a specific subclass, the inheritance is wrong.

The principle

Named after Barbara Liskov: if S is a subtype of T, you should be able to use an S anywhere a T is expected, and the program should still behave correctly. Subclasses must honour the promises their parent makes — not just match method signatures.

A violation

class Bird {
    void fly() { /* ...take off... */ }
}

class Ostrich extends Bird {
    @Override
    void fly() {
        throw new UnsupportedOperationException("ostriches can't fly");
    }
}

An Ostrich is-a Bird by the class hierarchy, but it breaks the promise. Any code that trusts "a Bird can fly" blows up when handed an Ostrich:

void migrate(Bird bird) {
    bird.fly();          // fine for a Sparrow... crashes for an Ostrich
}

The Ostrich is not substitutable for a Bird. That's the violation — the hierarchy lied about what an Ostrich can do.

The tell-tale sign

When a subclass overrides a method to throw, or to do nothing, or otherwise contradicts what the parent promised — that's the Liskov alarm. It means the "is-a" isn't really true for the behaviour in question.

The fix: model reality honestly

Not all birds fly, so fly() doesn't belong on Bird. Pull flying into its own capability (an interface — Lesson 12) that only the birds that fly implement:

class Bird {
    void eat() { /* all birds eat */ }
}

interface Flyer {
    void fly();
}

class Sparrow extends Bird implements Flyer {
    public void fly() { /* ...take off... */ }
}

class Ostrich extends Bird {
    // no fly() — and nothing pretends it can
}

Now nobody can write code that assumes every Bird flies. Only a Flyer promises fly(), and every Flyer keeps that promise. Substitutability restored.

Python comparison: same idea — a subclass that can't honour its parent's behaviour will surprise callers, duck typing or not.

Do this now (reason it out — answer in chat)

Classic case. A Square extends Rectangle:

class Rectangle {
    protected int width, height;
    void setWidth(int w)  { width = w; }
    void setHeight(int h) { height = h; }
    int area() { return width * height; }
}

class Square extends Rectangle {
    @Override void setWidth(int w)  { width = w; height = w; }   // keep it square
    @Override void setHeight(int h) { width = h; height = h; }
}

Answer these: A caller does rect.setWidth(5); rect.setHeight(4); then expects area() to be 20. What happens if rect is actually a Square? Why does that violate Liskov? Send me your reasoning.
